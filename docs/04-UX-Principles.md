# 04 · UX Principles

## Overview

Atlas is designed to support human thinking.

The interface should never compete with the work it is helping to organise.

Every interaction should feel direct, predictable and easy to reverse.

The interface exists to support building, not administration.

---

## The Builder Comes First

Atlas follows the builder.

It does not require work to be classified before it can be added.

It does not require certainty before relationships can be created.

Builders decide what matters.

Atlas adapts.

---

## Start Small

Atlas should always begin with the smallest possible action.

The first Thing.

The first Connection.

The first Attribute.

Complexity emerges naturally through use.

It is never presented up front.

---

## Never Interrupt Momentum

Adding something should be immediate.

Connecting Things should feel natural.

Description can always happen later.

Atlas should never interrupt creative flow to request information that is not immediately necessary.

---

## Every Gesture Has One Meaning

Interactions should always be predictable.

Dragging one Thing onto another always creates a Connection.

Dragging onto a Connection always inserts a Thing into that Connection.

The same gesture should never produce different results in different contexts.

---

## Build, Don't Configure

Builders should spend their time building knowledge.

Not configuring software.

Vocabulary emerges naturally through use.

Defaults should require little or no setup.

Configuration should remain optional.

---

## Teach Yourself Away

Atlas should become simpler as confidence grows.

New builders should be guided.

Experienced builders should barely notice the interface.

The software should gradually disappear behind the work.

---

## The Model Is More Permissive Than The Interface

The Core Model allows incomplete knowledge.

The interface encourages meaningful actions.

A Thing may exist without Attributes.

A Connection may exist without a label.

The interface should help builders enrich knowledge without making completeness a requirement.

---

## The Canvas Is A Workspace

The canvas is not a diagram.

It is a place for thinking.

Builders should feel comfortable adding, moving and connecting Things without concern for perfect organisation.

Structure emerges over time.

---

## Views Are Interpretations

Changing a View never changes the graph.

Views help builders explore knowledge from different perspectives.

Colour.

Shape.

Size.

Filtering.

Grouping.

These are interpretations of the graph, not changes to it.

---

## Vocabulary Emerges

Atlas does not begin with predefined vocabularies.

Attribute names emerge.

Attribute values emerge.

Connection labels emerge.

The builder's language becomes the language of Atlas.

---

## Remove Before Adding

Whenever a new concept is proposed, the first question should be:

> Can this emerge from the existing model?

If the answer is yes, no new concept should be introduced.

Simplicity is preserved by resisting unnecessary structure.

---

## Principles

Every design decision in Atlas should support these ideas.

- The builder comes first.
- Build before organising.
- Never interrupt momentum.
- Every gesture has one meaning.
- The model remains simple.
- Vocabulary emerges naturally.
- The graph is the truth.
- Views are interpretations.
- Remove before adding.
